# ADR 0001: Стратегия версионирования API

* **Статус**: Accepted
* **Дата**: 2026-01-22
* **Автор**: @winststreloc

## Контекст и проблема
С развитием проекта CoffeePeek возникла необходимость вносить ломающие изменения в API (например, изменение структуры ответа в профиле пользователя), сохраняя при этом поддержку старых клиентов. Требуется выбрать механизм версионирования, который будет единообразным для всех микросервисов.

## Рассмотренные варианты
1. **Версионирование через URL** (`/api/v1/users`):
   - Плюсы: Явно, легко кэшировать.
   - Минусы: "Засоряет" URL, требует дублирования маршрутов.
2. **Версионирование через заголовки** (`X-Api-Version: 1.0`):
   - Плюсы: Чистые URL, гибкость.
   - Минусы: Сложнее тестировать через браузер без плагинов.

## Решение
Выбрано  версионирование с упором на **заголовки**, реализованное через библиотеку `Asp.Versioning.Http`.

Детали реализации:
- Основной способ: заголовок `X-Api-Version`.
- Константы версий вынесены в `CoffeePeek.Shared.Infrastructure.Constants.ApiVersions` для переиспользования.
- Swagger настроен так, чтобы генерировать отдельные спецификации для каждой версии (v1, v2).

## Последствия
### Плюсы
* Возможность параллельного существования нескольких версий эндпоинта в одном контроллере (как сделано в `UsersController`).
* Прозрачное управление версиями через `Shared` библиотеку.
* Автоматическая генерация документации для новых версий.

### Минусы
* Необходимо следить за тем, чтобы новые контроллеры не забывали указывать `[ApiVersion]`.
