<objective>
Move the Yandex API key from the request URL query string to an HTTP request header.

Purpose: The API key is currently embedded in the URL as ?apikey={key}. ASP.NET Core's default HTTP client diagnostics log full outgoing request URLs in Development; Serilog, Sentry breadcrumbs, and OTLP traces may also capture the URL. Any log aggregation system will expose the key. The fix moves the key to DefaultRequestHeaders on the typed HttpClient (set once at DI registration, never appears in URLs again).

ASSUMPTION GATE: The Yandex Maps Geocoding API v1.x header authentication support is not confirmed in official documentation. The preferred fix (header approach) proceeds unless the executor can verify that Yandex 1.x does NOT support header auth. If the API returns 403 after the change, a fallback must be applied — see checkpoint below.

Output: DependencyInjection.cs adds client.DefaultRequestHeaders.Add("X-Yandex-API-Key", yandexOptions.ApiKey); YandexGeocodingService.cs URL no longer contains apikey= parameter.
</objective>

<task type="checkpoint:decision" gate="blocking">
  <decision>Confirm Yandex Geocoding API header authentication approach before applying the code change</decision>
  <context>
    The preferred fix moves the API key from the URL query string (?apikey=...) to an HTTP request header (X-Yandex-API-Key). This prevents the key from appearing in logs and traces.

    However, official Yandex Geocoding API v1.x documentation primarily documents the ?apikey query parameter. Whether the API accepts an HTTP header for authentication is unconfirmed (assumption A2 from research).

    Two approaches are available:

    Option A — Header approach (preferred):
      - DependencyInjection.cs: client.DefaultRequestHeaders.Add("X-Yandex-API-Key", yandexOptions.ApiKey)
      - YandexGeocodingService.cs: URL becomes "?geocode={encodedAddress}&format=json&results=1" (no apikey)
      - Risk: If Yandex returns 403, geocoding breaks for all shop suggestions

    Option B — URL isolation with log sanitization (safe fallback):
      - Keep apikey in URL as-is
      - Add Serilog request logging enricher to strip "apikey" query parameters from logged outgoing URLs
      - This prevents key exposure in logs while maintaining API compatibility
      - No risk of geocoding regression

    To verify Option A before committing: check https://yandex.com/maps-api/docs/geocoder/ for header auth support, or test a geocoding call with the header and no apikey query param.
  </context>
  <options>
    <option id="option-a">
      <name>Header approach (preferred)</name>
      <pros>API key never appears in any URL, log, trace, or browser history</pros>
      <cons>Yandex header auth unconfirmed — risk of 403 if not supported</cons>
    </option>
    <option id="option-b">
      <name>URL isolation with Serilog log sanitization (safe fallback)</name>
      <pros>Zero regression risk; API compatibility guaranteed</pros>
      <cons>Key still present in URL (not in logs); does not prevent key exposure in OTLP traces that capture raw URLs</cons>
    </option>
  </options>
  <resume-signal>Select: option-a or option-b. If option-a: confirm that Yandex API supports X-Yandex-API-Key header (or similar). If option-b: proceed with Serilog sanitization.</resume-signal>
