<objective>
Close BUG-05: today `CreateShopFromModerationService.CreateShopFromApprovedEventAsync` calls `shopRepository.Add(shop)` but never flushes the EF Core change tracker, so shops approved via the moderation pipeline are silently dropped. This plan injects `IUnitOfWork` into the service's primary constructor (preserving the existing parameter order — adding the new dependency directly before `ILogger`) and adds an `await unitOfWork.SaveChangesAsync(cancellationToken)` immediately after the `Add(shop)` call, mirroring the pattern used by `ModerationReviewApprovedHandler` in the Infrastructure consumers folder. A regression test in the new Shops.Application.Tests project locks the fix in.

Purpose: Prevent data loss in the cross-service approval workflow (Moderation -> Shops via RabbitMQ + Wolverine). The fix is constrained to the Application service per RESEARCH.md Option B — callers of `ICreateShopFromModerationService` should observe a fully completed creation, not a tracked-but-unsaved entity.

Output: One updated service file (constructor + one new awaited call) and one new test class with at least two Facts (SaveChangesAsync called once on the happy path; the duplicate-ModerationId guard still throws and never reaches the Add/Save block).
</objective>
