<objective>
Close BUG-03 (IDOR — Insecure Direct Object Reference): any authenticated user can today delete any review by guessing/observing its id. This plan adds an ownership check at the handler boundary, introduces a dedicated `ForbiddenException` (HTTP 403) in `CoffeePeek.Shared.Kernel`, threads the requesting user id from the controller into the command via the existing `[property: JsonIgnore]` convention, and locks the fix in with three regression tests in the new `CoffeePeek.Shops.Application.Tests` project.

Purpose: Eliminate a critical access-control gap (STRIDE: Tampering / ASVS V4 Access Control). The handler must reject non-owner delete attempts with HTTP 403 (not 401, which means "not authenticated"). The fix uses the same `BaseException`-driven HTTP status mapping that the rest of the codebase relies on, so `GlobalExceptionHandler` requires zero changes.

Output: One new exception type, one updated command record, one updated handler, one updated controller action, and one new test file with three Facts.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Gateway -> ShopsService | Authenticated DELETE request crosses the JWT-validated edge; the downstream service trusts `IUserContext` claims. |
| Controller -> Handler (Wolverine bus) | `DeleteReviewFromCoffeeShopCommand` is serialized through Wolverine; `[property: JsonIgnore]` ensures `RequestingUserId` cannot be supplied from the wire and is only ever set by the controller from the authenticated user context. |
| Handler -> Repository -> PostgreSQL | Soft delete is committed only after the ownership check passes. |
